## 1. 已接入公开源（≥2）

| 标识 | 名称 | 登录 | 方式 | 默认限速 |
|------|------|------|------|----------|
| `ccgp` | 中国政府采购网 | 否 | HTML 列表 `search.ccgp.gov.cn/bxsearch` + 详情页 | ≥4 秒/请求 |
| `cebpub` | 中国招标投标公共服务平台 | 否 | 旧公开列表 JSON + `ctbpsp.com` 当前详情 PDF.js | ≥1 秒/请求 |

实现位置：

- `backend/app/sources/ccgp_source.py`
- `backend/app/sources/cebpub_source.py`
- 注册：`backend/app/sources/registry.py`

思路参考开源 [TenderCrawler](https://github.com/ir-st/TenderCrawler)（MIT），本仓异步重写，未引入其反爬绕过逻辑。

---

CEBPUB 通过旧公开列表接口发现公告，将 `businessId` 映射为当前平台详情 UUID。Playwright 先从 PDF.js 文档对象读取内存字节并在本地解析，字节不可用时才读取文字层；有效扫描件使用本地 OCR。PDF.js 明确报告文件无效/损坏时快速终止，同轮不重试；临时未就绪才重建标签页重试一次。确定性失败在自动增量任务中冷却 24 小时，逐条手工重采和完整快照可绕过冷却。只有 URL UUID、公告标题与 PDF 身份校验通过才标记 `full`。验证码/安全验证标记为 `needs_human_verification`，不绕过；通用门户页和未经 ID/标题校验的旧接口响应不会进入公告正文。

当前详情站若仅返回空白 SPA 外壳，自动采集约 15 秒内失败，不再弹出临时浏览器。用户可在常用浏览器下载官方 PDF/HTML，再从采集结果详情导入；导入正文须通过标题或项目编号校验，上传二进制不长期保存。

## 2. 登录态源

| 标识 | 名称 | 登录 | 方式 |
|------|------|------|------|
| `login_portal` | 登录态招采门户 | **是** | Playwright 加载 storage state 访问检索/详情 |

### 合规流程

1. 用户运行 `scripts/login_source.ps1`（或 `python -m app.tools.login_init`）  
2. **可见浏览器**打开登录页，用户手动登录（自行处理验证码）  
3. 保存 `data/browser_states/login_portal_state.json`（**.gitignore**）  
4. 采集时使用该状态访问；检测到「请登录」等文案则判定失效  
5. 失效时**跳过登录源**，ccgp/cebpub **继续执行**  

### 配置项（`.env`）

```bash
LOGIN_SOURCE_ENABLED=true
LOGIN_SOURCE_HOME_URL=https://www.chinabidding.cn/
LOGIN_SOURCE_LOGIN_URL=https://www.chinabidding.cn/
LOGIN_SOURCE_SEARCH_URL=https://www.chinabidding.cn/search/?q={keyword}
LOGIN_SOURCE_STATE_FILE=login_portal_state.json
```

可改为您有合法权限的免费会员门户。详见 [操作文档.md](./操作文档.md)。

三个 URL 必须属于同一门户。检测到跨站配置时，`login_portal` 会被停用并返回明确原因；公开源不受影响。

登录源失败时，公开源仍继续执行（编排层已支持）。

## 3. 采集流程

```text
SearchTask 关键词/区域/日期
  → 各启用源 search(列表)
  → 列表二次过滤（关键词/区域/时间/是否招标类）
  → fetch_detail + extract_attachments
  → 详情二次过滤
  → 正文清洗与 PDF 页码保留
  → 规则快速提取 → 关键字段缺口 AI 抽取 → 原文证据校验 → 规则结果保底
  → 企业画像资格匹配与证据约束决策分析
  → 写入 TenderAnnouncement（data_mode=live）
  → TaskExecution 统计
```

检索深度分为：快速每源 8 条、标准每源 30 条（默认）、深度/完整快照每源最多 500 条。发现结果超过详情上限时必须返回 `detail_cap_skipped` 并标记 `truncated=true`，不得把已采集的前若干条表述成完整覆盖。正文 `document_hash` 与抽取指纹未变化时复用字段证据；模型连续超时两次后，本轮后续公告规则降级，不影响正文入库。

---

## 4. 字段与 data_mode

- `data_mode=live`：真实抓取  
- `data_mode=fixture`：仅测试/演示（界面金色标签）  
- `source_url`：尽量可回跳原始页  
- `attachment_links`：无附件则为 `[]`，不伪造  

---

## 5. 使用注意

1. **ccgp** 响应慢、易提示访问频繁 → 保持间隔 ≥4s，减少关键词/页数。  
2. 网站改版可能导致选择器失效 → 健康检查会提示；测试使用 mock HTML。  
3. 测试禁止对真实站点持续高频访问。  
4. 遵守各网站服务条款与 robots/访问规范，仅用于比赛演示与学习。

## 6. API

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/sources` | 源列表 |
| POST | `/api/sources/health-all` | 全部健康检查 |
| POST | `/api/sources/{name}/health` | 单源检查 |
| POST | `/api/tasks/{id}/execute` | 执行采集；支持 `trigger_type`、`search_depth`、`refresh_extraction` 与 `report_mode` |
| GET | `/api/announcements` | 结果列表 |
| GET | `/api/announcements/{id}` | 字段、页码证据、完整度、AI 分析与校正审计 |
| POST | `/api/announcements/{id}/recrawl` | 重新抓取详情、抽取并分析 |
| POST | `/api/announcements/{id}/import-detail` | 导入官方 PDF/HTML，校验后抽取并分析 |
| POST | `/api/announcements/{id}/reextract` | 仅用已保存正文重新抽取 |
| POST | `/api/announcements/{id}/analyze` | 按当前企业画像重新分析 |
| PATCH | `/api/announcements/{id}/fields` | 带原因保存人工校正 |
| POST | `/api/announcements/{id}/feedback` | 追加正确/错误质量反馈 |
| GET | `/api/announcements/quality-metrics` | 详情成功率、证据匹配率、缓存命中率和反馈准确率 |

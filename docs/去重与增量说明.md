# 去重与增量说明

> 阶段五 · FusionBid

---

## 1. 跨源去重

### 判定依据（组合）

1. 同源 `source_item_id`  
2. 完全相同 URL  
3. 项目/招标编号（正文与标题提取）  
4. 标准化标题 + 区域兼容 + 发布日期  
5. 标题相似度 ≥ 0.88 +（正文相似 / 附件名重叠 / 极高标题相似）  
6. 正文 `content_hash` 一致  

### 主记录保留优先级

1. 官方源优先：`ccgp` > `cebpub` > `login_portal`  
2. 正文更完整  
3. 附件更多  
4. 发布时间更完整  

### 合并元数据（不静默删除）

主记录字段：

- `related_urls`：相关来源 URL 列表  
- `related_sources`：被合并来源详情  
- `dedupe_reasons`：去重依据  
- `is_primary`：是否主记录  
- `project_code`：提取的项目编号

## 2. 增量推送

表：`delivery_histories`

| 规则 | 行为 |
|------|------|
| 任务下从未交付 | 作为**新增**输出 |
| 已交付且 `content_hash` 相同 | **跳过**（不重复推送） |
| 已交付但哈希变化 | 再次输出并标记 **「内容发生变化」** |
| 执行失败 | **不**写入 DeliveryHistory |

阶段五在采集执行 **success/partial** 后，将本次增量项的 `report_id` 记为 `execution_id`；阶段六 Word 生成后可改为报告 ID。

## 4. 执行 API 返回字段

`POST /api/tasks/{id}/execute` 额外包含：

- `duplicate_count` / `cross_source_merge_count`  
- `incremental_count` / `update_count` / `skipped_already_delivered`  
- `output_items`（含 `is_new` / `is_update` / `change_label`）  
- `dedupe_reasons`

# 去重与增量说明

> FusionBid v1.0.0 · 增量与完整快照语义

---

## 1. 跨源去重

### 判定依据（组合）

1. 同源 `source_item_id`  
2. 完全相同 URL  
3. 项目/招标编号（正文与标题提取）  
4. 标准化标题 + 区域兼容 + 发布日期只能形成候选，不能单独合并
5. 标题相似度 + 正文相似度或附件名重叠
6. 正文 `content_hash` 一致  

项目编号不同的记录不合并。项目编号相同但 `lifecycle_stage` 不同（机会公告、更正/澄清、结果公告、终止/废标）也不合并，只建立生命周期关联，避免后公告覆盖原公告。`procurement_method` 独立保存公开招标、询价、磋商、单一来源等采购方式，正文中的采购方式不得覆盖标题明确的结果/终止生命周期。

### 主记录保留优先级

1. 官方源优先：`ccgp` > `cebpub` > `login_portal`  
2. 正文更完整  
3. 附件更多  
4. 发布时间更完整  

### 合并元数据（不静默删除）

主记录字段：

- `related_urls`：相关来源 URL 列表  
- `related_sources`：被合并来源详情  
- `dedupe_reasons`：去重依据  
- `is_primary`：是否主记录  
- `project_code`：提取的项目编号

## 2. 增量推送

表：`delivery_histories`

| 规则 | 行为 |
|------|------|
| 任务下从未交付 | 作为**新增**输出 |
| 已交付且 `content_hash` 相同 | **跳过**（不重复推送） |
| 已交付但哈希变化 | 再次输出并标记 **「内容发生变化」** |
| 执行失败 | **不**写入 DeliveryHistory |

只有报告成功生成后才写入 DeliveryHistory；采集失败或报告生成失败均不标记为已交付。

## 3. 未去重完整快照

`report_mode=full_snapshot` 的语义固定为：重新抓取、不做跨源去重、不读取或写入 DeliveryHistory，并把本轮每个来源记录分别写入报告。后台公告表仍以稳定主键保存，完整快照的原始记录集合由本次执行直接交给报告生成器。

每个来源最多采集 500 条。达到上限时响应与报告标记 `truncated=true`，此时不得称为完整无截断结果。

## 5. 执行 API 返回字段

`POST /api/tasks/{id}/execute` 额外包含：

- `duplicate_count` / `cross_source_merge_count`  
- `incremental_count` / `update_count` / `skipped_already_delivered`  
- `output_items`（含 `is_new` / `is_update` / `change_label`）  
- `dedupe_reasons`  
- `report_mode` / `deduplicate` / `truncated`
- `detail_full_count` / `detail_metadata_count` / `detail_failed_count` / `detail_human_verification_count`
